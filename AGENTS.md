# AGENTS.md

## Project Overview

このリポジトリは「ポケモン技範囲考察ツール」のReactアプリ。

ユーザーが攻撃タイプを複数選択すると、防御側の全タイプ組み合わせに対して、選択中の攻撃タイプ群で「最大で何倍のダメージを与えられるか」を一覧できる。

主目的は限定できない。ユーザーが「この攻撃タイプ群でどの防御タイプに弱点を突けるか」「どの攻撃タイプを追加すれば弱点を突ける防御タイプが増えるか」を考察することもできるし、「逆にこの攻撃タイプ群に対し弱点を突かれないのはどの防御タイプか」を探すこともできる。よって、アプリ側が提示するのは「純粋な結果」であって、その結果に意味を与える表示や機能はこのアプリの担うところではない。

## Tech Stack

- React 18
- Create React App / `react-scripts`
- MUI v6
- データ定義:
  - `src/data/typeData.js`: タイプ表示名、色、文字色
  - `src/data/typeEffectiveness.js`: 防御タイプごとの攻撃タイプ倍率

## Domain Terms

- 攻撃タイプ: ユーザーが選択する、技のタイプ。
- 防御タイプ: 表の縦横で表す、相手ポケモンのタイプ。
- 単タイプ: 防御タイプ1つだけ。表では対角線上のマスとして扱う。
- 複合タイプ: 防御タイプ2つの組み合わせ。
- 最大倍率: 選択中の攻撃タイプ群のうち、その防御タイプに対して最も高いダメージ倍率。
- 弱点を突ける: 最大倍率が2倍以上になること。
- 弱点を突けない: 最大倍率が1倍以下になること。

## Current Functional Specification

### Attack Type Selection

- 18タイプの攻撃タイプボタンを表示する。
- タイプボタンをクリックすると選択/解除をトグルする。
- 複数タイプを同時に選択できる。
- `リセット`ボタンで選択攻撃タイプをすべて解除する。
- 選択済みタイプがある場合、表の上に現在選択中の攻撃タイプをタイプ色つきタグで表示する。

### Coverage Table

- 表の縦軸と横軸は防御側タイプを表す。
- 行ヘッダーと列ヘッダーには18タイプをタイプ色つきで表示する。
- 対角線上のマスは単タイプとして扱う。
- 対角線より片側のマスだけを複合タイプとして扱い、反対側は重複なので空欄/グレー表示にする。
- 攻撃タイプが未選択の場合、計算対象マスは `-` を表示する。
- 攻撃タイプが選択されている場合、各マスには選択攻撃タイプ群の中での最大倍率を表示する。

### Damage Multiplier Calculation

- 単タイプの倍率:
  - `typeEffectiveness[defenseType][attackType]`
- 複合タイプの倍率:
  - `typeEffectiveness[defenseType1][attackType] * typeEffectiveness[defenseType2][attackType]`
- マスに表示する値:
  - 選択中の全攻撃タイプについて上記倍率を計算し、その最大値を表示する。
- 表示対象の倍率:
  - `0`, `0.25`, `0.5`, `1`, `2`, `4`
- 小数表示は不要な末尾ゼロを削る。

### Table Color Semantics

現状の色分けは以下。

- `4`: 強い赤、白文字
- `2`: 薄い赤
- `1`: 白
- `0.5`: 薄い青
- `0.25`: 青
- `0`: 強い青、白文字
- 重複側の未使用マス: グレー

UI/UX改善時も、少なくとも「抜群」「等倍」「半減以下」「無効」「未使用マス」の違いが一目で分かることを優先する。

### Weak Coverage List

表の下には、現在選択中の攻撃タイプでは弱点を突けない防御タイプを表示する。

- 対象は最大倍率が `0.25`, `0.5`, `1` の防御タイプ。
- `0.25倍のタイプ`, `0.5倍のタイプ`, `1倍のタイプ` のグループに分ける。
- 各防御タイプについて、その相手に対して弱点を突ける追加候補の攻撃タイプを表示する。
- 追加候補は `4倍` と `2倍` に分けて表示する。
- 単タイプはタイプタグ1つ、複合タイプはタイプタグ2つで表示する。

### Additional Attack Type Summary

弱点を突けない防御タイプの一覧のさらに下に、追加候補攻撃タイプの単純集計を表示する。

- `4倍のタイプ`: 対象防御タイプ群に4倍を取れる攻撃タイプと件数。
- `2倍のタイプ`: 対象防御タイプ群に2倍を取れる攻撃タイプと件数。
- 件数は「その攻撃タイプを追加すると、対象防御タイプ群のうち何件に弱点を突けるか」の単純カウント。
- これは厳密な最適化ではなく、次に何タイプを加えると抜群範囲が広がりやすいかを見るための目安。

## UX Goals For Completion

今後UI/UXを改善して完成させるときは、以下を重視する。

- 初見でも「何を選べば何が分かるツールか」がすぐ分かる。
- 表・弱点リスト・追加候補サマリーの関係が自然に理解できる。
- 18x18表がPCでもスマホでも破綻しない。
- タイプタグや倍率の色は、ポケモンのタイプ感を保ちつつ可読性を優先する。
- 長大な弱点リストは、スキャンしやすく、必要に応じて折りたたみや要約ができる設計にする。
- 装飾よりも実用性を優先する。考察ツールとして、密度は保ちつつ迷子にならない画面にする。

## Implementation Guidelines

- 既存のタイプ順は `Object.keys(typeData)` を基準にする。
- タイプ相性の正確性は `typeEffectiveness.js` を基準にする。
- 同じ防御タイプ組み合わせを二重表示しない。
- 計算ロジックと表示ロジックは、可能なら分離してテストしやすくする。
- UIの改善であっても、最大倍率の定義を変えない。
- 「追加候補攻撃タイプの集計」は現状どおり単純集計として扱う。最小追加タイプ集合の探索など、別ロジックを入れる場合は別機能として明示する。
- MUIを既に使っているため、基本的にはMUIコンポーネントとReactの通常パターンに寄せる。
- 変更後は少なくとも `npm run build` で確認する。

## Current Known UX Problems

スクリーンショットと現状実装から見える改善候補。

- 表と下部リストが縦に長く、全体の情報構造がつかみにくい。
- 追加候補の重要度が視覚的に分かりづらい。
- 防御タイプごとの候補表示が小さく、読み取りに時間がかかる。
- スマホ幅では18x18表と候補リストの操作性が課題になりやすい。
